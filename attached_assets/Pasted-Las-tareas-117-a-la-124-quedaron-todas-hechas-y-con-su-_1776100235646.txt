2.	Problema a resolver
Actualmente, cuando un usuario reporta un error, el código se revisa caso por caso, usuario por usuario. Esto es insostenible porque:
Si hay un error, hay que arreglarlo para todos los usuarios, no uno por uno.
Puede haber 100 usuarios con el mismo error y habría que corregir 100 veces.
Lo que necesito
El código debe estar escrito de manera que:
Cuando se detecta un error, se corrige en UN SOLO LUGAR (la fuente central).
Esa corrección se aplica AUTOMÁTICAMENTE para TODOS los usuarios (nuevos y existentes).
NO se debe arreglar usuario por usuario manualmente.
NO se debe tocar la base de datos para parchear casos individuales (a menos que sea estrictamente necesario).
Ejemplo de lo que NO quiero
text
❌ Mal: Si el usuario A tiene error, escribo código especial solo para él.
❌ Mal: Si el usuario B tiene el mismo error, escribo otro código especial para él.
❌ Mal: Parchear en la base de datos usuario por usuario.
Ejemplo de lo que SÍ quiero
text
✅ Bien: Detecto el error en la lógica central (ej: asignación de posts a negocios).
✅ Bien: Corrijo la función o el endpoint en UN SOLO ARCHIVO.
✅ Bien: La corrección afecta automáticamente a TODOS los usuarios.
✅ Bien: Subo el cambio a producción y se arregla para todos.

Crear un sistema de desarrollo con las siguientes reglas OBLIGATORIAS:

REGLAS DE ARQUITECTURA:

1. Toda la lógica de negocio debe estar en funciones/endpoints CENTRALIZADOS.
2. NO se permite código que verifique "si es el usuario X, haz algo especial".
3. NO se permiten parches en la base de datos para corregir errores de lógica (solo para limpiar datos corruptos).
4. Cualquier corrección de error debe hacerse en UN SOLO LUGAR y afectar a TODOS los usuarios.

FLUJO DE CORRECCIÓN DE ERRORES:

1. Usuario reporta un error.
2. Se identifica la causa raíz en el código central.
3. Se corrige en UN SOLO ARCHIVO o FUNCIÓN.
4. Se despliega a producción.
5. AUTOMÁTICAMENTE todos los usuarios (nuevos y existentes) tienen la corrección.

LO QUE ESTÁ PROHIBIDO:

❌ Hacer "if user_id == 123: arreglar esto"
❌ Crear columnas especiales en base de datos para casos particulares
❌ Corregir el mismo error en 10 lugares diferentes
❌ Parchear manualmente usuario por usuario

LO QUE ES OBLIGATORIO:

✅ La lógica debe ser genérica y basada en reglas, no en IDs de usuarios
✅ Las correcciones benefician a todos automáticamente
✅ El código debe ser mantenible y escalable

PRUEBA OBLIGATORIA:

Antes de desplegar una corrección, probar con al menos 3 usuarios diferentes para verificar que funciona para TODOS.

Este Skill aplica para TODOS los desarrollos futuros y correcciones de errores.
