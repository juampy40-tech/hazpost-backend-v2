# Telegram Notifications — HazPost

## Regla de oro

**Lee este skill antes de tocar cualquier código de Telegram.** El sistema tiene dos niveles de configuración:

| Nivel | Dónde se guarda | Quién lo ve | Para qué |
|---|---|---|---|
| **Por usuario** | `users.telegram_bot_token`, `users.telegram_chat_id` | Solo ese usuario | Posts publicados / fallidos |
| **Admin global** | `app_settings` (`telegram_bot_token`, `telegram_chat_id`) | Solo admin | Niche insights, chatbot leads, autogeneración, fallback de alertas Meta |

---

## Esquema de DB

### Columnas en `users` (añadidas vía startup migration — NO en drizzle schema)

```sql
ALTER TABLE users
  ADD COLUMN IF NOT EXISTS telegram_bot_token TEXT,
  ADD COLUMN IF NOT EXISTS telegram_chat_id TEXT;
```

Estas columnas **no están** en `lib/db/src/schema/users.ts`. Se añaden en `artifacts/api-server/src/index.ts` (función `runStartupMigrations`). Siempre usar SQL crudo (`db.execute(sql\`...\``) para leerlas/escribirlas.

### `app_settings` (admin global — no cambia)

Claves: `telegram_bot_token`, `telegram_chat_id`

## Funciones del servicio

**Archivo**: `artifacts/api-server/src/services/telegram.service.ts`

### `getTelegramConfigForUser(userId: number): Promise<TelegramConfig | null>`
- Lee `telegram_bot_token` y `telegram_chat_id` de la tabla `users` usando SQL crudo.
- Retorna `null` si el usuario no tiene ambas columnas configuradas.
- Exportada para uso en tests o desde otros servicios.

### `notifyPostPublished(postId, caption, successPlatforms, failedPlatforms, userId?)`
- Si `userId` es provisto → usa el bot del usuario (`getTelegramConfigForUser`).
- Si `userId` no es provisto → usa el bot admin global.
- Si no hay config → silencia (no lanza error, no hace fallback a admin para posts de usuario).
- **Siempre pasar `post.userId`** desde el scheduler.

### `notifyPostFailed(postId, caption, platform, errorMessage, userId?)`
- Mismo comportamiento que `notifyPostPublished` en cuanto al lookup.
- **Siempre pasar `post.userId`** desde el scheduler.

### `notifyMetaTokenAlert(status, expiresAt, daysLeft, userId?)`
- Si `userId` provisto → intenta bot del usuario primero.
- Si usuario no tiene bot → fallback al bot admin global.
- Es la única función que tiene fallback explícito a admin.

### `notifyAutoGenerated(count, daysAhead)`
- Solo admin global (alerta de sistema, no de usuario).

### `notifyNicheInsights(analysis)`
- Solo admin global.

### `notifyChatLeadHot(conversationId, visitorName, conversationText)`
- Solo admin global.

### `sendTestNotification(botToken, chatId): Promise<{ ok, error? }>`
- Utilitaria pura: envía mensaje de prueba a credenciales dadas. No consulta DB.

### `detectChatId(botToken): Promise<{ chatId, firstName?, error? }>`
- Utilitaria pura: llama `getUpdates` de Telegram para obtener el ID del chat.

---

## Regla de lookup por función

```
notifyPostPublished / notifyPostFailed:
  userId provisto → getUserConfig → si null → SILENCIO (no fallback)

notifyMetaTokenAlert:
  userId provisto → getUserConfig → si null → getAdminConfig → si null → SILENCIO

notifyAutoGenerated / notifyNicheInsights / notifyChatLeadHot:
  siempre → getAdminConfig → si null → SILENCIO
```

## Endpoints de configuración (por usuario)

**Archivo**: `artifacts/api-server/src/routes/social/settings.ts`

Todos requieren `requireAuth` middleware.

| Método | Ruta | Descripción |
|---|---|---|
| `GET` | `/api/settings/telegram` | Lee la config del usuario autenticado. El token se enmascara como `"••••••••"`. |
| `PUT` | `/api/settings/telegram` | Guarda `telegram_bot_token` y/o `telegram_chat_id` para el usuario. Si se envía `"••••••••"`, el token no se actualiza. |
| `POST` | `/api/settings/test-telegram` | Envía mensaje de prueba usando las credenciales del usuario. El body puede sobrescribir las guardadas. |
| `POST` | `/api/settings/detect-telegram-chat-id` | Detecta el Chat ID llamando `getUpdates` con el bot del usuario. |

### Respuesta de GET y PUT `/api/settings/telegram`

```json
{
  "telegram_bot_token": "••••••••",
  "telegram_chat_id": "123456789",
  "configured": true
}
```

## Integración en el scheduler

**Archivo**: `artifacts/api-server/src/services/scheduler.service.ts`

Las llamadas deben pasar `post.userId` como quinto argumento:

```typescript
notifyPostPublished(postId, caption, successPls, failedPls, post.userId ?? undefined)
notifyPostFailed(postId, caption, platform, errorMessage, post.userId ?? undefined)
notifyMetaTokenAlert(status, expiresAt, daysLeft, account.userId ?? undefined)
```

## Frontend — settings.tsx

**Archivo**: `artifacts/social-dashboard/src/pages/settings.tsx`

- `telegramConfigured`, `telegramChatId` se cargan vía `useEffect` que llama `GET /api/settings/telegram`.
- `handleSaveTelegram` llama `PUT /api/settings/telegram` (no al endpoint global `PUT /api/settings`).
- `handleTestTelegram` y `handleDetectChatId` siguen llamando a los mismos endpoints POST, que ahora usan las credenciales del usuario autenticado.
