# Monitoreo y Alertas — HazPost

## Infraestructura de Monitoreo

| Canal | Qué monitorea | Dónde |
|-------|--------------|-------|
| **Logs pino** | Toda request HTTP + errores estructurados | `artifacts/api-server` |
| **Telegram** | Publicaciones exitosas/fallidas, auto-generación | `telegram.service.ts` |
| **Logs de consola** | Background jobs (imágenes, reels, scheduler) | Prefijo `[BG]`, `[Scheduler]`, etc. |
| **DB (publish_log)** | Historial de publicaciones por plataforma | `publish_log` table |

## 1. LOGGING ESTRUCTURADO (PINO)

### En route handlers — usar `req.log`
```typescript
// ✅ Correcto — pino estructurado, indexable
router.post("/approve", async (req, res) => {
  const log = req.log;
  try {
    const [post] = await db.update(postsTable)...
    log.info({ postId: post.id, userId: req.user!.userId }, "Post aprobado");
    res.json({ status: post.status });
  } catch (err) {
    log.error({ err, postId: req.params.id }, "Error aprobando post");
    res.status(500).json({ error: "Error interno" });
  }
});

// ❌ No usar console.log en handlers
console.log("Post aprobado:", postId);
```

### Qué incluir en un log de info
```typescript
log.info({
  userId: req.user!.userId,   // SIEMPRE — para trazabilidad por usuario
  businessId: post.businessId, // cuando aplica
  postId: post.id,
  action: "post_approved",    // acción descriptiva
}, "Post aprobado");
```

### Qué NUNCA loguear
```typescript
// ❌ NUNCA — expone datos sensibles
log.info({ token: req.cookies.hz_token }, "Request auth");
log.info({ password: req.body.password }, "Login attempt");
log.info({ apiKey: process.env.OPENAI_API_KEY }, "Calling OpenAI");

// ✅ En su lugar
log.info({ userId: req.user?.userId, hasToken: !!req.cookies.hz_token }, "Request auth");
```

### En background jobs — prefijo de módulo
```typescript
// ✅ Prefijo para identificar el origen en logs
console.error("[BG] Image generation error:", err);
console.warn("[Scheduler] Publish failed for post:", postId, err.message);
console.info("[LearningEngine] Startup extraction done.");
console.info("[Music] Biblioteca de música ya inicializada");
```

## 2. MANEJO DE ERRORES — PATRÓN ESTÁNDAR

### Route handler — try/catch explícito
```typescript
router.post("/:id/approve", requireAuth, async (req, res) => {
  try {
    const postId = Number(req.params.id);
    if (!postId || isNaN(postId)) {
      return res.status(400).json({ error: "ID de post inválido" });
    }

    const post = await getTenantPost(postId, req);
    if (!post) return res.status(404).json({ error: "Post no encontrado" });

    // ... lógica principal
    res.json({ success: true, post });

  } catch (err) {
    req.log.error({ err, postId: req.params.id }, "Error aprobando post");
    // Express maneja el 500 automáticamente con el error handler global
    throw err;
  }
});
```

### Mensajes de error — amigables para el usuario
```typescript
// ❌ Expone detalles internos
res.status(500).json({ error: err.message }); // "Cannot read property 'id' of undefined"

// ✅ Amigable, genérico para el usuario
res.status(500).json({ error: "Ocurrió un error al aprobar el post. Intenta de nuevo." });
// + log interno con detalle completo
```

### Background tasks — NUNCA bloquear por errores
```typescript
// ✅ Background: falla silenciosa con log
generateImagesForPostsBg(jobs).catch(err =>
  console.error("[BG] Image generation error:", err)
);

// ✅ Scheduler hooks: no bloquear el flujo principal
notifyPostPublished(post).catch(logger.warn);
```

## 3. ALERTAS TELEGRAM — CUÁNDO USAR

El bot de Telegram (`telegram.service.ts`) ya tiene 3 tipos de notificación:

| Función | Cuándo disparar |
|---------|----------------|
| `notifyPostPublished(post)` | Publicación exitosa (parcial o total) en `scheduler.service.ts` |
| `notifyPostFailed(post, error)` | Fallo total de publicación (Instagram + TikTok fallaron) |
| `notifyAutoGenerated(result)` | Auto-generación nocturna completada (cron 06:00 Bogotá) |

Para agregar nuevas alertas Telegram (ej: créditos agotados, nuevo usuario registrado):
```typescript
// En telegram.service.ts agregar nueva función
export async function notifyLowCredits(userId: number, remaining: number): Promise<void> {
  await sendTelegramMessage(
    `⚠️ <b>Créditos bajos</b>\nUsuario ID ${userId} tiene solo <b>${remaining} créditos</b> restantes.`
  );
}
```

## 4. MONITOREO DE USO — QUERIES SQL PARA EL DASHBOARD ADMIN

### Posts generados por día (últimos 30 días)
```sql
SELECT
  DATE(created_at AT TIME ZONE 'America/Bogota') as dia,
  COUNT(*) as posts_generados,
  COUNT(DISTINCT user_id) as usuarios_activos,
  COUNT(DISTINCT business_id) as negocios_activos
FROM posts
WHERE created_at > NOW() - INTERVAL '30 days'
GROUP BY DATE(created_at AT TIME ZONE 'America/Bogota')
ORDER BY dia DESC;
```

### Tasa de aprobación por negocio
```sql
SELECT
  b.name as negocio,
  COUNT(*) FILTER (WHERE p.status = 'published') as publicados,
  COUNT(*) FILTER (WHERE p.status = 'approved' OR p.status = 'scheduled') as aprobados,
  COUNT(*) FILTER (WHERE p.status = 'rejected') as rechazados,
  COUNT(*) FILTER (WHERE p.status = 'pending_approval') as pendientes,
  ROUND(
    100.0 * COUNT(*) FILTER (WHERE p.status IN ('approved', 'scheduled', 'published'))
    / NULLIF(COUNT(*) FILTER (WHERE p.status IN ('approved', 'scheduled', 'published', 'rejected')), 0),
    1
  ) as tasa_aprobacion_pct
FROM posts p
JOIN businesses b ON b.id = p.business_id
WHERE p.business_id IS NOT NULL
GROUP BY b.id, b.name
ORDER BY publicados DESC;
```

### Fallos de publicación (últimas 2 semanas)
```sql
-- publish_log.status: 'published' | 'failed'
SELECT
  pl.platform,
  COUNT(*) as intentos,
  COUNT(*) FILTER (WHERE pl.status = 'published') as exitosos,
  COUNT(*) FILTER (WHERE pl.status = 'failed') as fallos,
  ROUND(100.0 * COUNT(*) FILTER (WHERE pl.status = 'failed') / NULLIF(COUNT(*), 0), 1) as tasa_fallo_pct
FROM publish_log pl
WHERE pl.published_at > NOW() - INTERVAL '14 days'
GROUP BY pl.platform
ORDER BY fallos DESC;
```

### Usuarios más activos
```sql
SELECT
  u.email,
  u.plan,
  COUNT(p.id) as posts_totales,
  COUNT(p.id) FILTER (WHERE p.created_at > NOW() - INTERVAL '7 days') as posts_7d,
  s.credits_remaining as creditos_restantes
FROM users u
LEFT JOIN posts p ON p.user_id = u.id
LEFT JOIN subscriptions s ON s.user_id = u.id
GROUP BY u.id, u.email, u.plan, s.credits_remaining
ORDER BY posts_7d DESC
LIMIT 20;
```

## 7. PG POOL ERROR HANDLER — OBLIGATORIO EN PRODUCCIÓN

**Este fue el vector del crash de producción del 2026-04-19 (hazpost.app caído con crash loop).**

Cuando PostgreSQL termina una conexión idle (network blip, DB timeout, reinicio del servidor),
el pool de `pg` emite un evento `'error'` en el cliente. Sin un listener registrado, Node.js
trata ese evento como una excepción no capturada y **mata el proceso entero**.

### Patrón obligatorio en `lib/db/src/index.ts`

```typescript
export const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  connectionTimeoutMillis: 10000,
  idleTimeoutMillis: 30000,
  max: 10,
});

// CRÍTICO: sin este listener un idle-client error crashea el proceso entero
pool.on("error", (err) => {
  console.error("[pg-pool] idle client error — connection dropped unexpectedly:", err.message);
  // NO re-throw: el pool se recupera automáticamente creando nuevas conexiones
});
```

### Handlers globales de proceso en `artifacts/api-server/src/index.ts`

Agregar inmediatamente después de la validación del PORT (defensa en profundidad):

```typescript
process.on("unhandledRejection", (reason) => {
  logger.error({ reason }, "[process] Unhandled Promise Rejection");
});

process.on("uncaughtException", (err) => {
  logger.error({ err }, "[process] Uncaught Exception — process will exit");
  process.exit(1); // salida limpia con código != 0 para que el runtime reinicie
});
```

### Por qué `pool.on('error')` es suficiente vs re-throw

- El pool de `pg-pool` crea nuevas conexiones automáticamente cuando una muere
- Hacer re-throw causaría el mismo crash que queríamos evitar
- El handler solo necesita loguear — el pool se recupera solo

## 6. CHECKLIST DE MONITOREO AL AGREGAR NUEVA FUNCIONALIDAD

Cuando implementes un nuevo feature, preguntarte:

- [ ] ¿Hay un log de info cuando la operación principal tiene éxito?
- [ ] ¿Hay un log de error (con detalle) cuando falla?
- [ ] ¿Los mensajes de error al usuario son amigables (no exponen stack trace)?
- [ ] ¿Las tareas background usan `.catch(logger.warn)` para no bloquear el flujo?
- [ ] ¿Si algo puede fallar silenciosamente (webhook, email, Telegram), hay un log de warn?
- [ ] ¿Si la operación afecta créditos o publicaciones, debería disparar una alerta Telegram?
